# Agent 项目迁移提示词

## 目标

把当前仓库迁移为一个适合 Agent 持续协作、可验证、可维护的项目。

你要做的不只是“改页面”，而是把项目整理成一个让 Agent 能持续工作的代码库：

- 明确项目边界和职责分层
- 把可复用能力沉淀成技能、文档和约定
- 把容易出错的逻辑改成可验证、可回归的流程
- 把“做了什么、为什么这么做、做了会怎么样”写清楚

## 迁移原则

1. 先保可用，再做统一。
2. 先落规则，再谈风格。
3. 先收敛核心能力，再扩展表现层。
4. 每次修改都要可解释。
5. 每次修改都要可验证。

## 当前项目语境

这个仓库已经具备以下基础：

- 有 `AGENTS.md` 作为 Agent 行为规范入口
- 有 `.github/skills/` 作为项目级技能入口
- 有 `.vscode/mcp.json` 作为工作区 MCP 配置入口
- 有 `docs/` 作为开发说明、迁移说明和用户说明入口

迁移时要延续这些入口，而不是再造一套分散的新约定。

## 迁移重点

### 1. 项目结构迁移

把项目从“页面驱动”迁移为“能力驱动”：

- 代码只负责实现
- 技能负责方法
- 文档负责解释
- MCP 负责外部能力接入

### 2. Agent 工作流迁移

让 Agent 可以持续在这个仓库里工作，而不是每次都重新理解上下文：

- 重要约定写进 `docs/`
- 可复用能力写进 `.github/skills/`
- 外部工具写进 `.vscode/mcp.json`
- 行为规则写进 `AGENTS.md`

### 3. 体验与设计迁移

如果项目仍然保留前端界面，要统一这些体验：

- 视觉层级清楚
- 卡片、边框、留白、字号一致
- 深色模式可读
- 导航、搜索、统计、阅读辅助等基础交互稳定
- 保留个性化，不把所有页面做成同一种模板

## 输出要求

每次做完迁移，都要按下面三段输出：

### 做了什么

说明你具体修改了哪些结构、行为、文档或配置。

### 为什么这么做

说明这个改动解决了什么问题，或者避免了什么风险。

### 做了会怎么样

说明这次迁移会对后续开发、维护、测试、Agent 协作带来什么变化。

## 执行顺序

1. 先读当前文档、技能和配置，确认已有约定。
2. 找出当前迁移的最薄弱环节。
3. 先修会导致错误、混乱或误导的地方。
4. 再做结构收口、体验收口和文档收口。
5. 最后验证构建、浏览器行为和文档一致性。

## 验证标准

迁移完成前，至少满足以下条件：

- 构建可通过
- 核心交互可用
- 文档能解释当前状态
- Agent 能根据现有技能和配置继续工作
- 关键改动有可复现的验证结果

## 文档要求

迁移期间要同步维护这些文档：

- `docs/DEVELOPMENT.md`：当前开发入口，简洁说明“做了什么 / 为什么 / 会怎么样”
- `docs/DEVELOPMENT_LOG.md`：按时间记录详细演化
- `docs/TUTORIAL.md`：讲清楚如何使用和扩展
- `docs/USER_GUIDE.md`：讲清楚日常怎么操作和配置

## 记忆要求

如果本轮迁移形成了稳定约定，要把它记录到仓库记忆里，后续 Agent 可以直接复用。

## 你在工作时要优先做的事

- 先找当前控制逻辑，不要先猜
- 先做最小能验证的修改
- 先保证功能，再追求更漂亮的表达
- 先让后续 Agent 看得懂，再让用户看得顺

## 结束标准

当且仅当以下内容都达成时，才算这次 Agent 项目迁移完成：

- 项目结构清晰
- 核心能力稳定
- 文档完整可读
- 技能和 MCP 可用
- 交互体验可持续维护
- 修改理由和结果都写清楚
